# 状態・実績 UI 再整理計画

## 方針

`StatusDetailPanel` にある独自のアビリティ取得機能は廃止する。
衣装確認モードの解放はスキルツリーへ統合し、同じ用途の成長要素を「アビリティ」と「スキル」の二系統で管理しない。

役割は次のように分離する。

- 状態ボタン: 現在のキャラクター状態を閲覧する
- スキルボタン: スキルの取得、解放条件、装備・有効化を管理する
- 実績表示ボタン: 訓練・戦闘の累計と内訳を閲覧する
- 衣装確認設定: 取得済みスキルによって利用可能になったモードを選択する

状態画面から能力を購入・取得する導線は置かない。

## 衣装確認能力のスキル統合

現在の `ConditionalOutfitPrompt` と `HiddenOutfitPrompt` は、主人公用スキルツリーノードの付与効果へ移す。
実装時は、衣装確認モードの解放効果を表すデータを `SkillData` または `SkillTreeNodeData` に追加する。

候補フィールドは次のとおり。

```text
unlockedOutfitPromptModes:
- Conditional
- Hidden
```

取得済み主人公ノードから利用可能モードを再構築し、`GameManager.CanUseScheduledEventOutfitPromptMode()` はその結果を参照する。
既存の `OutfitPromptAbilitySet` は、移行中は現在モードの保持と旧セーブ読込に限定し、解放状態の正本にはしない。

モードの役割は次のようにする。

- `Always`: 初期状態で利用可能
- `Conditional`: 対応スキル取得後に利用可能
- `Hidden`: 対応スキル取得後に利用可能

スキル取得とモード選択は別の操作とする。
スキル取得直後に現在モードを強制変更せず、新しく選べる選択肢だけを増やす。
現在選択中のモードが利用不能になった場合は `Always` へ戻す。

モード選択 UI は状態画面の能力取得欄には置かない。
初期実装ではスキル画面の取得済み便利スキル詳細、または予定・衣装設定に専用の「衣装確認設定」欄を置く。
単なる解放済みスキル一覧のボタンとモード切り替えボタンを兼用しない。

## 状態画面の整理

`StatusDetailPanel` は閲覧専用にする。
主人公・ヒロイン切り替えは維持し、最初に必要な現在値を簡潔に表示する。

主人公表示:

- 名前
- HP / 最大 HP、MP / 最大 MP
- 攻撃、防御、素早さ
- 好感度に関係しない主人公固有値
- 所持金
- 現在衣装、現在の衣装確認モード
- 使用中の戦闘スキル、訓練スキルの要約

ヒロイン表示:

- 名前
- 好感度と好感度段階
- HP / 最大 HP、MP / 最大 MP
- 攻撃、防御、素早さ
- 現在衣装
- 使用中の戦闘スキル、訓練スキルの要約

一覧が長くなる情報は常時展開しない。
取得済み全スキル、購入済み全アイテム、詳細なBuff/Debuff、訓練・敵別実績は別画面または詳細ボタンから表示する。

削除対象となる状態画面の要素:

- `abilityListParent` を使う能力一覧
- 能力取得確認パネル
- 能力取得ボタン
- 能力ごとの解放条件・必要ポイント表示
- `StatusAbilityData` のResources自動読込
- 状態画面からの衣装確認モード解放処理

## 実績画面の整理

既存の `StatusProgressPanel` を実績表示の正規画面として使用する。
状態画面の「実績」ボタンから開く導線は維持するが、状態画面本文へ実績値を重複表示しない。

最初の画面は全体概要とする。

- 累計訓練回数
- 主人公 LP 消費回数
- ヒロイン LP 消費回数
- 同時 LP 消費回数
- 総モンスター撃破数
- 使用可能なスキルポイントと累計獲得量（集計可能になった段階で追加）

詳細切り替え:

1. 全体
2. 訓練カテゴリー別
3. 訓練別
4. 敵別

各詳細では、項目名、現在値、関連する解放条件の有無を表示する。
未記録値は 0 とし、データがないカテゴリーを大量に空表示しない。
スキルツリーの解放条件表示から該当実績へ移動できる拡張は維持する。

## データ移行

現行の関連データは次のとおり。

- `StatusAbilityData`
- `StatusAbilityKind`
- `StatusAbilityEffectType`
- `SaveData.unlockedStatusAbilityIds`
- `GameManager.unlockedStatusAbilityIds`
- `OutfitPromptAbilitySet.canUseConditionalMode`
- `OutfitPromptAbilitySet.canUseHiddenMode`

移行は段階的に行う。

1. スキル／ノードへ衣装確認モード解放効果を追加する
2. 既存の衣装確認アビリティと同等の主人公ノードを作成する
3. 利用可否判定を取得済みスキルツリーから再構築する
4. 状態画面から能力取得 UI を外す
5. 旧能力IDを対応ノード取得へ変換するセーブ移行を必要に応じて追加する
6. 参照がなくなった後に `StatusAbilityData` 系と `unlockedStatusAbilityIds` を削除する

移行途中でも、既存セーブで選択中の衣装確認モードが不正にならないよう `Always` へのフォールバックを維持する。
旧セーブ互換を不要と判断した場合でも、Nullや未定義Enum値で停止しない読込処理は残す。

## 実装順

1. 衣装確認モード解放用のスキルデータ構造を決める
2. Conditional / Hidden用の主人公スキルツリーノードを追加する
3. 解放判定をスキルツリー基準へ切り替える
4. モード選択 UI の配置先を決めてコード接続する
5. `StatusDetailPanel` の能力取得関連コードとUIを撤去する
6. 状態画面の表示項目とレイアウトを整理する
7. `StatusProgressPanel` の全体・カテゴリー・訓練・敵表示を整理する
8. セーブ／ロード、主人公・ヒロイン切り替え、スキル取得直後を回帰確認する

## 実装状況

衣装確認モードのスキルツリー統合と、状態画面からの取得UI停止まで実装済み。

- `SkillTreeNodeData` に衣装確認モード解放効果を追加
- 主人公ノード「衣装確認・条件表示」「衣装確認・非表示」を追加
- 取得済みノードの詳細ボタンからモードを使用／解除可能
- 状態画面は解放状況と現在モードの読み取り表示だけを残す
- SaveData version 16以前の解放フラグを対応ノードへ移行

旧 `StatusAbilityData`、旧セーブフィールド、Scene上の旧UIオブジェクトの物理削除は、移行確認後の整理工程として残す。

画像追加はこの整理の必須条件にしない。
既存UI部品の再配置とテキスト表示を先に完成させる。
